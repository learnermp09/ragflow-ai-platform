Day01 Learnings:
---------------
i) pyproject.toml is the primary configuration file for a Python project. 
It defines the project's metadata and its ++dependency++ requirements.
                                          
ii) How does uv.lock help over time?

Suppose your project is deployed today.

One year later, you clone the repository onto a new computer and run:

uv sync

Because uv.lock is present, uv installs the same versions that were
originally tested, instead of the latest available versions. 
This allows you to recreate the original development environment even
if hundreds of newer package releases exist.

2. write code for config.py  
-------------------------
This file actually picks all variables from .env file using pydantic models and functions
"""
Application configuration.

All configurable values are defined here and loaded from environment
variables or the local `.env` file using Pydantic Settings.
"""

from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    """Application settings."""

    # Project
    PROJECT_NAME: str = "RAGFlow AI Platform"  # default value
    PROJECT_VERSION: str = "1.0.0"
    ENVIRONMENT: str = "development"

    # Hugging Face
    HUGGINGFACEHUB_API_TOKEN: str  # There is no default value. This variable must exist.
    LLM_REPOSITORY: str = "deepseek-ai/DeepSeek-R1-0528"
    PROVIDER: str = "auto"

    # Embeddings
    EMBEDDING_MODEL: str = (
        "sentence-transformers/all-mpnet-base-v2"
    )
    EMBEDDING_DEVICE: str = "cpu"
    NORMALIZE_EMBEDDINGS: bool = False

    # Document processing
    PDF_DIRECTORY: str = "backend/us_census"
    CHUNK_SIZE: int = 1000
    CHUNK_OVERLAP: int = 200

    '''
        CHUNK_SIZE: int = 1000

        If CHUNK_SIZE=500 exists inside .env

        then settings.CHUNK_SIZE becomes 500
    '''    
    # Vector store
    VECTOR_STORE_DIRECTORY: str = "backend/vectorstore"

    # LLM generation
    MAX_NEW_TOKENS: int = 512
    TEMPERATURE: float = 0.0
    REPETITION_PENALTY: float = 1.03

    # Logging
    LOG_LEVEL: str = "INFO"
    LOG_DIRECTORY: str = "logs"

    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
        extra="ignore",
    )


# Shared settings instance
settings = Settings()

The key idea is that the field names in Settings act as a schema. 
BaseSettings uses that schema to look up environment variables with matching names, 
convert them to the declared Python types, validate required values, and expose 
them as attributes on the single settings object that the rest of your application imports.

Q : Should there be any use of 
custom exception module in constants.py and config.py?

Ans:
This is an excellent architectural question. The answer is **no**—and there is an important software engineering reason behind it.

## Short Answer

| File           | Use `logger`? | Use `RAGFlowException`? |
| -------------- | ------------- | ----------------------- |
| `config.py`    | ❌ No          | ❌ No                    |
| `constants.py` | ❌ No          | ❌ No                    |

---

# Why not?

## 1. `constants.py`

This file should contain **only constant values**.

Example:

```python
APP_NAME = "RAGFlow AI Platform"

LOG_FORMAT = "..."

API_VERSION = "v1"
```

Nothing in this file executes.

Nothing can fail.

There is nothing to log.

There is nothing to catch.

Therefore

* no logger
* no exception class

---

## 2. `config.py`

This one is slightly more interesting.

Our `config.py` is responsible for

```
Load .env
↓

Create settings object
↓

Return configuration
```

Now imagine this happens

```
HUGGINGFACEHUB_API_TOKEN missing
```

Pydantic itself raises a validation error.

That is exactly what we want.

Example

```
ValidationError

HUGGINGFACEHUB_API_TOKEN
Field required
```

This is much clearer than

```
RAGFlowException

Configuration Error
```

because during startup we actually want the application to fail immediately.

---

# More importantly...

Look at the dependency graph.

Today

```
config.py
```

is imported by

```
logger.py
```

because

```python
settings.logging.level
```

comes from `config.py`.

Now imagine `config.py` imports `logger`.

```
config.py
↓

imports logger

↓

logger.py

↓

imports config.py
```

You get

```
Circular Import
```

which is one of the most common problems in Python projects.

---

# Same with exceptions

Imagine

```
config.py

↓

imports exception.py
```

Later

```
exception.py

↓

imports logger.py

↓

imports config.py
```

Again

```
Circular dependency
```

---

# Professional Architecture

The dependency direction should always be one-way:

```
constants.py
        │
        ▼
config.py
        │
        ▼
logger.py
        │
        ▼
exception.py
        │
        ▼
services/
        │
        ▼
rag.py
        │
        ▼
FastAPI
```

Notice

```
config.py
```

never depends on

```
logger.py
```

or

```
exception.py
```

Only the reverse.

This keeps the architecture clean and avoids circular imports.

---

# Where should `RAGFlowException` be used?

Starting from modules that perform real work, such as:

```
create_vectorstore.py
```

```
rag.py
```

```
document_loader.py
```

```
embedding.py
```

```
FastAPI routes
```

```
database.py
```

These modules can fail due to I/O, networking, or runtime conditions, so wrapping exceptions and logging them there is appropriate.

---

# Final Architecture for This Project

| Module                  | Logger           | Exception         | Reason                                                            |
| ----------------------- | ---------------- | ----------------- | ----------------------------------------------------------------- |
| `constants.py`          | ❌                | ❌                 | Static values only                                                |
| `config.py`             | ❌                | ❌                 | Bootstrap configuration; let Pydantic report configuration errors |
| `logger.py`             | Uses `config.py` | ❌                 | Initializes logging                                               |
| `exception.py`          | ❌                | Defines exception | Centralized custom exception                                      |
| `create_vectorstore.py` | ✅                | ✅                 | Runtime operations                                                |
| `rag.py`                | ✅                | ✅                 | Runtime operations                                                |
| `app.py`                | ✅                | ✅                 | Application orchestration                                         |
| `FastAPI` routes        | ✅                | ✅                 | Request handling                                                  |

This layered dependency structure is what I'd use for a production AI backend because it keeps the foundation (`constants.py` and `config.py`) simple, deterministic, and free of circular dependencies.

Q: I am seeing two files in vectorstore folder: 1. index.faiss and index.pkl
Ans:  Yes — this is exactly what you should expect. It indicates that your FAISS vector store was successfully created and persisted to disk.

Here's what those two files contain:

File	Purpose	Required
index.faiss	Stores the FAISS vector index (the numerical embeddings and search index).	✅ Yes
index.pkl	Stores the LangChain metadata, including the mapping between FAISS vector IDs and the original Document objects (text, metadata, source, page, etc.).	✅ Yes
How they work together
PDF Documents
      │
      ▼
Text Chunks
      │
      ▼
Embeddings (768-dimensional vectors)
      │
      ▼
FAISS Index
      │
      ├── index.faiss   ← Vector index
      │
      └── index.pkl     ← Document metadata

When you later execute:

vectorstore = FAISS.load_local(...)

LangChain:

Loads index.faiss into memory.
Loads index.pkl.
Reconstructs the complete FAISS vector store object.
Associates each vector with its corresponding document and metadata.

Without index.pkl, FAISS could still perform nearest-neighbor searches, 
but it would not know which document or text chunk each vector corresponds to.

2. index.pkl

This file is created by LangChain, not FAISS.

It stores everything that FAISS does not know about.

Remember:

FAISS only stores vectors.

It has no idea that vector #57 came from

backend/us_census/acsbr-015.pdf
Page 10

or that the text was

"The American Community Survey collects demographic data..."

That information is stored inside index.pkl.

What is inside index.pkl?

It is a serialized (pickled) Python object containing:

Document Objects

Each Document looks like

Document(
    page_content="The American Community Survey ...",

    metadata={
        "source": "backend/us_census/acsbr-015.pdf",
        "page": 12
    }
)

There are hundreds of these.

It also stores the mapping

Vector #0
↓

Document 0

-------------------

Vector #1
↓

Document 1

-------------------

Vector #2
↓

Document 2

This mapping is called the Docstore.

Why do we need it?

Suppose FAISS says

Nearest vector = #127

Without index.pkl, we would only know

Vector #127

But we would have no idea

which PDF?
which page?
what text?

So LangChain opens

index.pkl

and retrieves

Vector #127

↓

Document(
    page_content="American Community Survey ...",

    metadata={
        source="backend/us_census/acsbr-015.pdf",
        page=18
    }
)

This is exactly what your retriever returns.

"""
Response cleaner.

This module removes internal reasoning blocks and unnecessary
whitespace from Large Language Model (LLM) responses before they
are returned to the client.
"""

import re

from app.core.logger import logger


def clean_response(response: str) -> str:
    """
    Clean the raw response returned by the language model.

    This function removes:
    - <think>...</think> reasoning blocks
    - Leading and trailing whitespace

    Parameters
    ----------
    response : str
        Raw response generated by the language model.

    Returns
    -------
    str
        Cleaned response suitable for the end user.
    """

    if not response:
        logger.warning("Received an empty response from the language model.")
        return ""

    logger.info("Cleaning language model response.")

    # Temporary debugging logs
    logger.info("Raw response length: %d", len(response))
    logger.debug("Raw response:\n%s", response)

    cleaned_response = re.sub(
        pattern=r"<think>.*?</think>",
        repl="",
        string=response,
        flags=re.DOTALL,
    )

    cleaned_response = cleaned_response.strip()

    # Temporary debugging logs
    logger.info("Cleaned response length: %d", len(cleaned_response))
    logger.debug("Cleaned response:\n%s", cleaned_response)

    logger.info("Language model response cleaned successfully.")

    return cleaned_response

"""

This module creates and configures the FastAPI application for the
RAGFlow AI Platform.

from contextlib import asynccontextmanager

from fastapi import FastAPI

from app.api.routes import router
from app.core.config import settings
from app.core.logger import logger

async def lifespan(app: FastAPI):

    logger.info("RAGFlow AI Platform started successfully.")

)

class ProjectSettings(BaseModel):
    """General project information."""

    name: str = APP_NAME
    version: str = APP_VERSION
    environment: str = "development"


class APISettings(BaseModel):
    """REST API configuration."""

    version: str = API_VERSION
    prefix: str = API_PREFIX


class HuggingFaceSettings(BaseModel):
    """Hugging Face Hub configuration."""

    api_token: str
    provider: str = "auto"


class EmbeddingSettings(BaseModel):
    """Embedding model configuration."""

    model_name: str = EMBEDDING_MODEL
    device: str = "cpu"
    normalize_embeddings: bool = False


class DocumentSettings(BaseModel):
    """Configuration for document loading and chunking."""

    pdf_directory: str = DOCUMENT_DIRECTORY
    chunk_size: int = 1000
    chunk_overlap: int = 200


class VectorStoreSettings(BaseModel):
    """Vector store configuration."""

    directory: str = VECTORSTORE_DIRECTORY


class LLMSettings(BaseModel):
    """Large language model configuration."""

    repository: str = LLM_REPOSITORY
    task: str = LLM_TASK
    max_new_tokens: int = 512
    temperature: float = 0.0
    repetition_penalty: float = 1.03


class LoggingSettings(BaseModel):
    """Application logging configuration."""

    level: str = "INFO"
    directory: str = LOG_DIRECTORY


class AppSettings(BaseSettings):
    """Application settings loaded from the environment."""

    HUGGINGFACEHUB_API_TOKEN: str

    project: ProjectSettings = ProjectSettings()
    api: APISettings = APISettings()
    embedding: EmbeddingSettings = EmbeddingSettings()
    document: DocumentSettings = DocumentSettings()
    vectorstore: VectorStoreSettings = VectorStoreSettings()
    llm: LLMSettings = LLMSettings()
    logging: LoggingSettings = LoggingSettings()

    @property
    def huggingface(self) -> HuggingFaceSettings:
        """Return the Hugging Face configuration."""

        return HuggingFaceSettings(
            api_token=self.HUGGINGFACEHUB_API_TOKEN,
        )

    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
        extra="ignore",
    )


settings = AppSettings()
