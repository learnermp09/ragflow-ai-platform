Day01 Learnings:
---------------
i) pyproject.toml is the primary configuration file for a Python project. 
It defines the project's metadata and its ++dependency++ requirements.
                                          
ii) How does uv.lock help over time?

Suppose your project is deployed today.

One year later, you clone the repository onto a new computer and run:

uv sync

Because uv.lock is present, uv installs the same versions that were
originally tested, instead of the latest available versions. 
This allows you to recreate the original development environment even
if hundreds of newer package releases exist.

2. write code for config.py  
-------------------------
This file actually picks all variables from .env file using pydantic models and functions
"""
Application configuration.

All configurable values are defined here and loaded from environment
variables or the local `.env` file using Pydantic Settings.
"""

from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    """Application settings."""

    # Project
    PROJECT_NAME: str = "RAGFlow AI Platform"  # default value
    PROJECT_VERSION: str = "1.0.0"
    ENVIRONMENT: str = "development"

    # Hugging Face
    HUGGINGFACEHUB_API_TOKEN: str  # There is no default value. This variable must exist.
    LLM_REPOSITORY: str = "deepseek-ai/DeepSeek-R1-0528"
    PROVIDER: str = "auto"

    # Embeddings
    EMBEDDING_MODEL: str = (
        "sentence-transformers/all-mpnet-base-v2"
    )
    EMBEDDING_DEVICE: str = "cpu"
    NORMALIZE_EMBEDDINGS: bool = False

    # Document processing
    PDF_DIRECTORY: str = "backend/us_census"
    CHUNK_SIZE: int = 1000
    CHUNK_OVERLAP: int = 200

    '''
        CHUNK_SIZE: int = 1000

        If CHUNK_SIZE=500 exists inside .env

        then settings.CHUNK_SIZE becomes 500
    '''    
    # Vector store
    VECTOR_STORE_DIRECTORY: str = "backend/vectorstore"

    # LLM generation
    MAX_NEW_TOKENS: int = 512
    TEMPERATURE: float = 0.0
    REPETITION_PENALTY: float = 1.03

    # Logging
    LOG_LEVEL: str = "INFO"
    LOG_DIRECTORY: str = "logs"

    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
        extra="ignore",
    )


# Shared settings instance
settings = Settings()

The key idea is that the field names in Settings act as a schema. 
BaseSettings uses that schema to look up environment variables with matching names, 
convert them to the declared Python types, validate required values, and expose 
them as attributes on the single settings object that the rest of your application imports.

Q : Should there be any use of 
custom exception module in constants.py and config.py?

Ans:
This is an excellent architectural question. The answer is **no**—and there is an important software engineering reason behind it.

## Short Answer

| File           | Use `logger`? | Use `RAGFlowException`? |
| -------------- | ------------- | ----------------------- |
| `config.py`    | ❌ No          | ❌ No                    |
| `constants.py` | ❌ No          | ❌ No                    |

---

# Why not?

## 1. `constants.py`

This file should contain **only constant values**.

Example:

```python
APP_NAME = "RAGFlow AI Platform"

LOG_FORMAT = "..."

API_VERSION = "v1"
```

Nothing in this file executes.

Nothing can fail.

There is nothing to log.

There is nothing to catch.

Therefore

* no logger
* no exception class

---

## 2. `config.py`

This one is slightly more interesting.

Our `config.py` is responsible for

```
Load .env
↓

Create settings object
↓

Return configuration
```

Now imagine this happens

```
HUGGINGFACEHUB_API_TOKEN missing
```

Pydantic itself raises a validation error.

That is exactly what we want.

Example

```
ValidationError

HUGGINGFACEHUB_API_TOKEN
Field required
```

This is much clearer than

```
RAGFlowException

Configuration Error
```

because during startup we actually want the application to fail immediately.

---

# More importantly...

Look at the dependency graph.

Today

```
config.py
```

is imported by

```
logger.py
```

because

```python
settings.logging.level
```

comes from `config.py`.

Now imagine `config.py` imports `logger`.

```
config.py
↓

imports logger

↓

logger.py

↓

imports config.py
```

You get

```
Circular Import
```

which is one of the most common problems in Python projects.

---

# Same with exceptions

Imagine

```
config.py

↓

imports exception.py
```

Later

```
exception.py

↓

imports logger.py

↓

imports config.py
```

Again

```
Circular dependency
```

---

# Professional Architecture

The dependency direction should always be one-way:

```
constants.py
        │
        ▼
config.py
        │
        ▼
logger.py
        │
        ▼
exception.py
        │
        ▼
services/
        │
        ▼
rag.py
        │
        ▼
FastAPI
```

Notice

```
config.py
```

never depends on

```
logger.py
```

or

```
exception.py
```

Only the reverse.

This keeps the architecture clean and avoids circular imports.

---

# Where should `RAGFlowException` be used?

Starting from modules that perform real work, such as:

```
create_vectorstore.py
```

```
rag.py
```

```
document_loader.py
```

```
embedding.py
```

```
FastAPI routes
```

```
database.py
```

These modules can fail due to I/O, networking, or runtime conditions, so wrapping exceptions and logging them there is appropriate.

---

# Final Architecture for This Project

| Module                  | Logger           | Exception         | Reason                                                            |
| ----------------------- | ---------------- | ----------------- | ----------------------------------------------------------------- |
| `constants.py`          | ❌                | ❌                 | Static values only                                                |
| `config.py`             | ❌                | ❌                 | Bootstrap configuration; let Pydantic report configuration errors |
| `logger.py`             | Uses `config.py` | ❌                 | Initializes logging                                               |
| `exception.py`          | ❌                | Defines exception | Centralized custom exception                                      |
| `create_vectorstore.py` | ✅                | ✅                 | Runtime operations                                                |
| `rag.py`                | ✅                | ✅                 | Runtime operations                                                |
| `app.py`                | ✅                | ✅                 | Application orchestration                                         |
| `FastAPI` routes        | ✅                | ✅                 | Request handling                                                  |

This layered dependency structure is what I'd use for a production AI backend because it keeps the foundation (`constants.py` and `config.py`) simple, deterministic, and free of circular dependencies.

Prompt Management

This is an important module because every interaction with the LLM should originate from a single location. We want prompt engineering to be centralized rather than scattering prompt strings across the codebase.

Why do we need prompts.py?

As the project grows, we'll likely use different prompts for:

Question Answering
Document Summarization
Chat
Citation Generation
Follow-up Questions
Evaluation

If prompts are hardcoded inside rag.py, updating them
later becomes difficult. 
A centralized module makes maintenance much easier.

==============================
"""
Prompt templates used throughout the RAGFlow AI Platform.

Keeping prompts in a single module makes them easier to maintain,
reuse, and improve without modifying application logic.

  "I don't have enough information in the provided documents."
- Do not create or assume facts.
- Keep the answer concise and accurate.

Context:
{context}

Question:
{input}

Answer:
"""
)

============backend/app.py
import sys

from app.core.config import settings
from app.core.constants import APP_NAME, APP_VERSION
from app.core.exception import RAGFlowException
from app.core.logger import logger


def initialize_application() -> None:
    """
    Initialize the backend application.
    """

    logger.info("Starting application...")

    logger.info("Project Name : %s", APP_NAME)

    logger.info("Version      : %s", APP_VERSION)

    logger.info("Environment  : %s", settings.project.environment)

    logger.info("Configuration loaded successfully.")

    logger.info("Application initialized successfully.")


def main() -> None:
    """
    Application entry point.
    """

    try:
        initialize_application()

    except Exception as error:
        logger.exception("Application startup failed.")
        raise RAGFlowException(error, sys)


if __name__ == "__main__":
    main()
